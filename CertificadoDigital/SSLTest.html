Utilitário SSLTest

A adoção de certificado digital e protocolo SSL para realizar integração segura e criptografada entre servidores de aplicação tem gerado para os projetos de desenvolvimento e equipes de operação uma quantidade variada de problemas no tocante a conectividade. Devido a quantidade de possibilidade de erro que vão desde um certificado expirado até mesmo uma URL mal formada, se torna um cenário difícil de se diagnosticar e evitar erros. Tomando por base este cenário, a SUPST criou este utilitário para realizar uma bateria de testes em todos os componentes utilizados no processo de handshake SSL na infra-estrutura que utilizam esta tecnologia, mais especificamente SSL mútuo (SSL 2-way). Além de ser um componente genérico que atende a todas as demandas de SSL, foi incluído no utilitário testes de acionamento aos serviços corporativos mantidos pela SUPST a saber: (senha-rede, captcha, sms, etc) adotarem SSL mútuo.

Requisitos para Uso do Utilitário de Teste

  • Servidor de destino com SSL ativo.
  • Para testes de SSL mútuo é necessário par de chaves com a senha da chave privada e o certificado de cliente.
    • Normalmente estes artefatos são armazenados em apenas uma arquivo do tipo .pfx (PKCS12) ou .jks (Java JKS)
  • Cadeia de certificados com a Autoridade Certificadora que emitiu o certificado do servidor.
    • Normalmente a cadeia é representada com todos os certificados de autoridade desde a AC raiz até a AC que emitiu o certificado do servidor.
  • Programa de teste
    • JAR executável SSLTest [1]
      • ZIP contendo um exemplo completo de estrutura para execução de teste SSL com o servidor de desenvolvimento do serviço senha-rede. [2]
  • Java 1.6 ou posterior

Glossário

Termo
 Significado
KeyStore
Arquivo que contém a chave privada, chave pública e o certificado. É possível também conter a cadeia de autoridades que emitiu o certificado em questão.
Este arquivo pode possuir uma senha para ser aberto, e esta senha geralmente é a mesma senha da chave privada contida nele.
Existem vários tipos de KeyStore e cada um armazena seus dados de forma diferente, podendo utilizar algoritmos de criptografia diferentes. A plataforma Java implementa um padrão próprio JKS, já o padrão PKCS12 é implementado pela empresa RSA Security e é amplamente utilizado nas diversas plataformas, inclusive java.
TrustStore
Arquivo que contém os certificados das autoridades confiáveis.
Para o uso apenas leitura deste arquivo, não há necessidade de informar senha para abrí-lo. No caso de validação se um determinado certificado foi emitido por uma autoridade contida neste arquivo, não há necessidade de informar a senha. Para editá-lo já há a necessidade de utilizar senha.
Este arquivo é um arquivo do tipo "KeyStore", sem a necessidade de armazenar tipos de chaves criptográficas, apenas os certificados das autoridades confiáveis.

Configuração

O programa de teste funciona apoiado em um arquivo de configuração de nome "profiles.properties" o qual precisa ficar na mesma pasta do programa de teste. A partir das configurações definidas pelo usuário o programa realiza os testes desejados.

Abaixo descrevemos as possíveis configurações que podem ser utilizadas no arquivo "profiles.properties":


Variável Significado
keyStoreFile
Uso obrigatório para testes de SSL mútuo.
Apontamento para o arquivo que contenha o par de chaves, o certificado e opcionalmente a cadeia de certificados de ACs do mesmo.
Ao realizar teste de autenticação mútua, é obrigatório a configuração da senha da chave privada (keyStorePassword).
Ex:
  • Apontamento local
keyStoreFile=keystore-cliente.jks
  • Apontamento referencial
keyStoreFile=/opt/certs/keystore-cliente.jks
keyStorePassword
Uso obrigatório para testes de SSL mútuo.
Senha da chave privada contida no arquivo de par de chaves
keyStoreType
Existem alguns tipos de keyStore aceitos por padrão pela JVM, como por exemplo JKS, PKCS12 e JCEKS.
É comum nos ambientes java trabalhar com o tipo JKS, já nos outros ambientes, microsoft por exemplo, o uso de PKCS12 é mais comum.
O tipo padrão JVM é JKS, então, caso seu keyStore for deste tipo, esta configuração é desnecessária.
Ex:
  • keyStoreType=JKS
  • keyStoreType=PKCS12
trustStoreFile
Arquivo que contém os certificados das autoridades confiáveis.
Ex:
  • Apontamento local
trustStoreFile=truststore.jks
  • Apontamento referencial
trustStoreFile=/opt/certs/truststore.jks
trustStoreType
Idem ao KeyStoreType
timeOut
Tempo que a conexão irá esperar uma resposta do servidor configurado com SSL.
Tempo em milisegundos
host
Endereço do servidor a ser testado.
Não é permitido colocar o protocolo, como por exemplo http://google.com, apenas o DNS do serviço, como por exemplo google.com.
Ex:
  • Ambiente de Desenvolvimento do Serviço Senha-Rede
desautenticacao.senharede.servicoscorporativos.serpro
  • Google
google.com
port
Porta no servidor que estará com o SSL ativado. A porta 80 geralmente NÃO utiliza SSL, sendo a porta padrão de SSL a 443.
Este valor pode ser alterado para que outros serviços sejam configurados de forma diferente no mesmo servidor.
Pode-se, por exemplo, ativar a porta 443 para SSL simples e a porta 8443 para SSL mútuo.
sslDebug
Valor passado para a JVM na variável "javax.net.debug".
A JVM irá gerar um buffer com as informações necessárias para o handshake do SSL. Este buffer é descarregado no arquivo ssldebug.txt no mesmo diretório da aplicação de teste.
Verificar a documentação da JVM para saber os valores a serem utilizados.
Ex:
  • Ativando o debug SSL
sslDebug=ssl

Exemplo do arquivo profiles.properties

Com o aplicativo de testes e os arquivos keystore.jks e truststore.jks no mesmo diretório, o arquivo de configuração ficaria assim: 
keyStoreFile=keystore.jks
keyStorePassword=changeit
trustStoreFile=truststore.jks
timeOut=10000
sslDebug=ssl
host=desautenticacao.senharede.servicoscorporativos.serpro
port=8082
Com esta configuração, ao executar o programa de teste ele irá tentar realizar um handshake no endereço desautenticacao.senharede.servicoscorporativos.serpro, na porta 8082, com timeout definido em 10 segundos, utilizando como certificado de cliente um certificado que esteja no arquivo keystore.jks e verificar se o certificado do servidor foi emitido por uma das autoridades confiáveis que estão no arquivo truststore.jks.

Executando o Teste

O comando para executar o teste é: 
java -jar ssltest.jar
A saída da execução deste teste é esta: 
Truststore: truststore.jks
Teste de conexão SSL com o servidor [desautenticacao.senharede.servicoscorporativos.serpro] na porta [8082]
Iniciando conexão... OK!
Iniciando handshake... OK!
Testes de SSL executados com sucesso!
Como a configuração de debug está ativada, no mesmo diretório foi gerado um arquivo de nome ssldebug.txt que contém as informações trocadas entre as máquinas no momento do handshake.

Utilitário de Teste do Serviço de Autenticação Corporativo Senha-rede

É possível utilizar o programa de testes acima para também testar o serviço web-service do senha-rede.

Necessidades

  • As mesmas necessidades para testar o SSL.
  • Host / Porta de um dos ambientes do serviço corporativo
    • Desenvolvimento: desautenticacao.senharede.servicoscorporativos.serpro : 8082
    • Homologação: homautenticacao.senharede.servicoscorporativos.serpro : 8444
    • Produção: autenticacao.senharede.servicoscorporativos.serpro : 443
  • Um usuário/senha válido no senha-rede e/ou um certificado digital de um usuário válido no senha-rede
  • Um certificado digital de máquina habilitado no serviço corporativo senha-rede
  • Um perfil de sistema válido no senha-rede

Configurações para o Teste SOAP

Variável Significado
soapUser
Usuário senha-rede para realizara autenticação. É o CPF do usuário.
soapPass
Senha do usuário do senha-rede
soapCert
Certificado Digital de um usuário do senha-rede
soapSystem
Perfil do sistema em que o usuário deve estar habilitado.
